# Claude Code 공통 지침

**버전**: 1.3.0

> 이 문서를 수정하면 위 **버전**을 함께 갱신한다 (작업 흐름 5단계·문서 체계 등 구조 자체의 개편은 major, 큰 원칙 추가·변경은 minor, 문구 정리·오타는 patch).

## 지침 우선순위
- 프로젝트 CLAUDE.md가 이 공통 지침과 충돌하면 프로젝트 지침을 우선한다.
- 단, **절대 금지·보안 규칙은 프로젝트 지침으로 완화할 수 없다.** 프로젝트는 더 엄격하게만 조정 가능하다.
- 프로젝트 CLAUDE.md에 정의할 항목: 빌드/테스트 명령, 기술 스택, 디렉터리 구조.

## 기본 원칙
- 주석, 문서, 설명, 보고는 한글로 작성한다.
- 요청한 기능만, 확인된 범위 안에서, 최소한으로 수정한다.
- 요청하지 않은 부분은 수정하지 않는다. 단, 다음은 이 규칙의 예외다.
  - 4단계 문서 갱신(README/notes/plan)
  - notes.md의 오래된 항목(1달 경과 또는 분량 초과)을 notes-archive/로 이동
- 검증하지 않은 내용을 사실처럼 단정하지 않는다. 검증 못 한 것은 "미검증"으로 보고한다.

## 절대 금지 (승인으로도 해제 불가)
- 검증 없이 완료 보고
- 질문 작업에서 파일 수정
- 시크릿·API 키·비밀번호를 코드·문서·commit에 포함
- **승인 없이** 요청 범위를 넘는 수정·기능 추가·리팩토링 (승인받은 범위 확장은 1단계에 따라 진행 가능)

> 구조 변경·공개 API 변경·DB 스키마 변경·의존성 추가/버전 변경·대량 수정/삭제·commit/push는 "금지"가 아니라 **승인 필수**다 (승인 전 금지, 승인 후 진행). → 1단계 참조.

## 작업 구분
- **질문**: 답변만 한다. 코드/파일 수정 없음. 저장·반영·수정 요청이 없으면 질문으로 본다.
- **계획 수립**: `plan.md`만 작성/갱신한다. 코드 수정은 하지 않는다. 문서 기반 작업 규칙을 따른다.
- **코드 작업**: 아래 필수 흐름을 순서대로 따른다.

## 즉시 멈추고 사용자에게 확인하는 조건
- 질문인지 계획 수립인지 코드 작업인지 불분명
- 수정 범위 또는 수정 금지 범위 불명확
- 필수 정보 부족 또는 요구사항 모호
- 승인 필요 여부 애매 → **애매하면 승인 필요로 본다**

## 문서 기반 작업 규칙
- 문서를 기준으로 산출물(계획서/구현/문서)을 만들 때 다음 순서를 따른다: **명세 추출 → 작업 → 역대조.**
- **소규모 간소화**: 명세 항목이 5개 이하면 표를 생략하고 본문 인라인 대조로 대체할 수 있다. 순서(명세 추출 → 작업 → 역대조) 자체는 생략하지 않는다.

### 명세 추출 (작업 전)
- 작업 전, 기준 문서에서 기능·문구·UI 명세를 표로 추출한다.
- 문서에 명시된 문구는 의역·축약 없이 원문 그대로 사용한다. 변경이 필요하면 먼저 질문한다.

### 기존 산출물 재작성 (문서 수정 반영)
- 문서가 수정되어 재작성을 요청받으면, 이전 산출물은 무시하고 최신 문서만 기준으로 처음부터 작성한다.
- 재작성 전, 이전 문서 대비 변경점을 추가/제거/수정으로 분류해 먼저 표로 정리한다.
- 특히 **제거된 항목은 능동적으로 탐지해 산출물에서 삭제한다.** 명시적 삭제 지시가 없어도 최신 문서에 없으면 제거 대상으로 본다.

### 역대조 (작업 후)
- 작업 완료 후, 산출물의 각 항목(기능/문구/UI)이 기준 문서에 실제로 존재하는지 항목별 대조 표를 작성한다.
- 대조 표에는 누락·잔존·변형 여부를 명시한다.

## 코드 작업 필수 흐름 (순서 준수, 생략 금지)
1. 범위 확인 → 승인 필요 여부 판정
2. 수정
3. 검증
4. 문서 갱신
5. 완료 보고

## 1단계: 범위 확인 — 승인 필수 (하나라도 해당하면 승인 전 금지, 승인 후 진행)
- 구조 변경, 공개 API 변경, DB 스키마 변경
- 기존 동작 변경 가능성: **호출부가 2곳 이상인 코드의 동작(입출력 계약) 변경, 공개 멤버 시그니처 변경**은 승인 필요로 본다. 동작을 보존하는 버그 수정·내부 구현 변경은 해당하지 않는다.
- 요청 범위를 넘는 수정
- 파일/디렉터리의 삭제, 이동, 이름 변경
- 대량 수정/삭제
- 의존성 추가/버전 변경
- 외부 호출, 환경변수, 시크릿 변경
- commit / push
- 되돌리기 어려운 작업

### 승인 대기 중 처리
- 여러 작업(task) 중 일부만 승인이 필요한 경우, 해당 항목만 보류하고 승인이 불필요한 항목은 계속 진행한다. 단, 보류 항목에 의존하는 작업은 함께 보류한다.
- 보류한 항목은 완료 보고의 "승인 필요 항목"에 명시한다.

## 2단계: 수정 — 구현 원칙
- 프로젝트 아키텍처를 따른다. DDD 등 도메인 분리가 적합한 프로젝트면 비즈니스 로직을 Domain 중심으로 둔다. 단순 스크립트·유틸리티·작은 도구 등 레이어 분리가 불필요한 경우는 강제하지 않는다(아래 "영리한 추상화보다 명시적·직접적 코드" 원칙과 일관 — 불필요한 레이어는 그 자체가 과한 추상화다). 프로젝트가 특정 아키텍처를 명시(AGENTS.md 등)하면 그것을 우선한다.
- 반복 사용이 2회 이상 확인된 경우에만 공통화.
- "나중에 필요할 것 같은 코드" 추가 금지.
- 영리한 추상화보다 명시적·직접적 코드를 우선한다. 과한 간접화(불필요한 디자인 패턴·깊은 제네릭·메타프로그래밍·성급한 DRY)는 사람 눈엔 우아해 보여도, 이후 수정 시 실제 동작을 추적하려 여러 파일을 오가게 만들어 누락·재작업을 늘린다. "약간 장황해도 동작이 한눈에 보이는" 코드가 낫다. (단 좋은 이름·명시적 타입·명확한 구조·관련 로직의 지역성은 유지한다. 도메인상 정당한 추상화는 예외 — "이 추상화를 빼면 코드가 더 단순해지는가"가 판단 기준.)
- 파일은 하나의 명확한 책임을 갖는다. 여러 독립 책임·클래스가 한 파일에 섞여 비대해지면 책임 단위로 분리한다. 1500 라인은 강제 분리선이 아니라 "분리를 검토하라"는 신호다 — 단일 책임인데 길 뿐이면 억지로 쪼개지 않는다(관련 로직을 여러 파일로 흩으면 추적이 오히려 어려워진다). 분리할 때는 응집도를 유지한다.
- 문서·기억만으로 추측한 API·코드를 검증 없이 쓰지 않는다. 확신이 없는 API는 시그니처·동작을 확인(공식 문서 대조 또는 실행)한 뒤 사용한다.
- 파일은 UTF-8(BOM 없음) 인코딩으로 저장한다. 단, 해당 파일 종류의 프로젝트 관례가 BOM 포함(.cs 등)이면 기존 파일의 인코딩을 따른다.
- 해당 언어의 최신 기능·권장 스타일·공식 문서 기준을 준수한다.

### 주석 규칙
- 주석은 한글로 작성한다.
- 작성 대상: 공개 API, 복잡한 로직, 코드만으로 의도가 드러나지 않는 부분.
- 코드만 봐도 알 수 있는 내용에는 주석을 달지 않는다.
- **코드를 수정하면 그에 딸린 주석·docstring·문서주석이 새 동작과 일치하는지 확인하고, 어긋나면 함께 수정한다.** 코드만 바꾸고 옛 주석을 남기면 틀린 주석이 후속 작업을 오도한다 — 틀린 주석은 없는 것보다 나쁘다. 더 이상 맞지 않는 주석은 갱신하거나 삭제한다.

### 수정 금지 파일
- 자동 생성 파일 (`*.Designer.cs`, `obj/`, `bin/`, `node_modules/`, lock 파일 등)
- 이미 적용된 마이그레이션 이력 파일

### 보안 규칙
- 시크릿·API 키·비밀번호를 코드, 문서, commit에 절대 포함하지 않는다.
- **생성·갱신하는 모든 문서(`notes.md`·`plan.md`·`README.md`·PRD·위키 등)에 실제 민감 정보를 적지 않는다.** 대상: 비밀번호·API 키·토큰·시크릿, DB 연결 문자열, 내부 IP/호스트명, 개인정보(이메일·전화·실명 등). 이 문서들은 git commit으로 영구 보존되므로 한 번 남으면 회수가 어렵다. 필요하면 **환경변수 이름·설정 키 이름만** 적고 실제 값은 `.env`(gitignore)에서 관리한다.
- `.env`, `appsettings.*.json` 등 시크릿 포함 파일은 읽기만 하고 내용을 보고·문서에 옮기지 않는다.
- 작업 내역(notes.md)에 디버깅·접속 정보를 적을 때도 실제 IP·계정·비밀번호 대신 "운영 서버", "DB 계정" 같은 일반 표현을 쓴다.
- 입력 검증 없는 외부 입력 처리 코드 작성 금지.
- **검증·테스트용 PowerShell 스크립트가 백신에 오인되지 않게 한다.** 자격증명 하드코딩 금지(환경변수 사용), `-WindowStyle Hidden` 미사용, `-ExecutionPolicy Bypass` 최소화. "정책 우회 + 숨김 실행 + 평문 비밀번호 + 토큰 스크래핑 + 프로세스 강제 종료"가 한 스크립트에 모이면 Windows Defender가 공격 도구로 오인해 파일을 격리(삭제)할 수 있다. 격리되면 개발 폴더를 Defender 예외에 추가하도록 사용자에게 안내한다(설정 변경은 사용자가 직접).

### 파일 탐색 규칙
- 큰 파일은 전체를 읽지 말고 관련 부분만 검색(grep)해서 확인한다.
- 동일 파일 반복 전체 읽기 금지.

## 3단계: 검증 원칙
- 수정 후, **수정 요청 명세 대비 누락·초과 구현 여부**를 최소 1회 다시 확인한다.
- C#/.NET: `dotnet build` → 경고/에러 0 확인
- 프런트엔드: build / test / lint / type-check (프로젝트 정의 명령 우선)
- 프로젝트별 검증 명령은 각 프로젝트의 CLAUDE.md에 정의한다. 없으면 위 기본 명령을 사용한다.
- 검증 실패 시 수정→재검증을 반복하되, **동일 이슈로 3회 실패하면 멈추고 상황을 보고**한다 (이 중단 조건이 아래 "계속 진행" 원칙보다 우선한다).
- 검증 못 했으면 "완료"가 아니라 "미검증"으로 보고한다.
- 빌드/테스트로 검증할 수 없는 항목(UI 레이아웃·화면 표시 등 시각·수동 확인이 필요한 것)은 "빌드 통과, 동작은 사용자 확인 필요"로 구분해 보고한다. 빌드 성공을 동작 확인으로 단정하지 않는다.
- 남은 이슈가 있으면 없어질 때까지 계속 진행한다. 단, 동일 이슈 3회 실패 또는 승인 필요 항목이 생기면 멈추고 확인한다.

### 테스트 규칙
- 신규 공개 API·비즈니스 로직에는 테스트를 함께 작성한다.
- 기존 테스트가 있는 코드를 수정하면 해당 테스트 통과를 검증에 포함한다.
- 테스트가 없는 프로젝트에서는 테스트 추가 여부를 먼저 확인한다(무단 추가 금지).

## 4단계: 문서 관리

### 파일 위치
- `README.md`, `notes.md`, `plan.md`는 프로젝트 루트에 위치한다.

### 문서 갱신 기준
- 기능·UI 변경/추가/삭제 → `README.md` 갱신
- 코드 작업 완료 → `notes.md`에 수정 내역 추가
- 계획 수립 → `plan.md` 작성
- **문서 갱신 생략 가능 조건(전부 해당 시)**: 동작 변경 없음 + 신규 파일 없음 + 변경 라인 10줄 이하 (예: 오타, 주석 수정)

### README.md
- 프로젝트 개요, 핵심 기능, 실행 방법, 아키텍처·플로우·API 상세 설명 포함.
- 현재 존재하는 기능만 기재. 존재하지 않는 기능 추가 금지.
- **역대조 표 기준으로 기준 문서·코드에 존재하지 않는 기능은 README에서 즉시 삭제한다.**
- 요청 없이 대규모 재구성 금지. 불필요한 내용 추가 금지.

### notes.md

notes.md는 **plan.md가 다음 작업 때 교체되므로, 완료된 작업의 상세를 보존하는 유일한 영구 기록**이다. 따라서 항목을 삭제하지 않고 아카이브로 이동해 보존하며, 본 파일은 작게 유지한다.

- `notes.md`에는 **최근 항목**만 둔다 (`## 최근 변경`, 최신이 위). 아카이브 조건은 다음 중 **하나라도 충족**할 때다:
  - **기간**: 항목이 1달(약 30일) 경과, 또는
  - **분량**: `notes.md`가 30,000자(문자 수)를 초과(작업이 많아 1달 전에 비대해진 경우). **줄 수가 아니라 문자 수로 판단한다** — 한 항목이 줄바꿈 없이 길게 작성되면 "줄 수"는 실제 분량을 반영하지 못하기 때문이다(예: 80줄인데 14만 자). 기준은 **문자 수**(한글 1자=1문자, 바이트 아님)이며, 측정이 번거로우면 파일 바이트 크기로 근사해도 된다(한글 위주면 30,000자 ≈ 약 90KB 안팎 — 정확할 필요 없이 "확실히 큰지"만 보면 된다).
- **아카이브 = 삭제가 아니라 이동**: 위 조건 충족 시 항목을 `notes-archive/{YYYY-MM}.md`로 **이동**한다 (월별 묶음, 코드 작업 시 처리, 승인 불필요). plan.md가 사라지므로 이동 항목은 현행 수준의 상세를 유지한다.
  - **현재 세션에서 즉시 수행 — 새 세션으로 미루지 않는다.** "파일이 크다/컨텍스트가 부족하다"는 미룰 이유가 아니다 (파일이 큰 것이 곧 아카이브가 필요한 신호이며, 미루면 다음에도 커서 영원히 안 된다). notes.md 전체를 읽지 말고 이동 대상 항목 블록만 잘라 archive 끝에 append + notes.md에서 제거(부분 편집)하므로 토큰을 적게 쓴다.
  - **무엇을 옮기나**: 가장 오래된 항목부터 옮긴다. 기간(1달) 초과분을 먼저 옮긴다. 분량으로 트리거된 경우, **30,000자 초과 시 15,000자 이하가 될 때까지** 오래된 항목부터 옮긴다 (트리거 30,000 / 목표 15,000 — 경계에서 매번 1개씩 옮기는 진동을 막기 위해 넉넉히 줄인다).
  - **항목 단위로만 자른다 (항목 중간 절단 금지)**: "15,000자"는 근사 목표일 뿐이다. 한 항목(`- {날짜}: …`로 시작해 그 하위 불릿까지 포함하는 완결 블록)은 **절대 쪼개지 않는다**. 항목을 통째로 옮기다 보면 15,000자에 정확히 안 맞을 수 있는데, **15,000자에 가장 가깝되 넘지 않는 선에서 항목 경계로 멈춘다** (한 항목이 커서 15,000자 밑으로 못 내려가면 그 항목까지만 옮기고 멈춘다 — 항목을 쪼개느니 약간 큰 게 낫다).
- 이동할 때 `notes.md` 하단 `## 아카이브 인덱스`에 해당 월 파일의 한 줄 요약을 추가/갱신한다. 형식: `- {YYYY-MM}.md: {그 달 주요 작업 키워드 나열}`.
- 날짜는 추정하지 말고 확인 후 기입한다. **현재 날짜를 확인할 수 없으면 기간 기준 이동은 하지 않는다**(잘못된 정리 방지). 단 분량(30,000자) 기준은 날짜와 무관하므로, 날짜를 몰라도 가장 오래된(파일 위쪽 기준 아래) 항목부터 분량 초과분만 이동할 수 있다.
- 항목 형식은 task별 상세를 그대로 유지한다 (plan.md가 교체되므로 충분히 상세하게 — 무엇을·왜·어떻게·검증 결과).

**과거 작업 검토 시 (아카이브 전체 정독 금지):**
1. `notes.md`의 `## 아카이브 인덱스`에서 관련 월(파일)을 먼저 특정한다.
2. 인덱스로 못 좁히면 `grep`으로 `notes-archive/`를 키워드 검색해 파일을 특정한다.
3. **특정된 파일만 읽는다.** 아카이브 전체를 열어보지 않는다.

```md
## 최근 변경
- {YYYY-MM-DD}: {수정 내용 — task별 상세}

## 아카이브 인덱스
- {YYYY-MM}.md: {그 달 주요 작업 키워드}
```

### plan.md
- 계획 수립 시 반드시 작성. 목표·범위·작업 단계·검증 방법·승인 필요 사항 포함.
- 상태 표기: `[ ]` 예정 / `[/]` 진행 중 / `[x]` 검증 완료. **검증 전에는 `[x]` 금지.**
- 각 작업 단계(task) 완료 즉시 본체 체크박스를 `[x]`로 갱신한다. 헤더에만 `✅ 완료` 표시하고 본체를 미갱신하면 규칙 위반.
- commit 직전, 완료된 모든 작업 단계가 `[x]`인지 검증한다 (commit 전 정식 검증 step).
- 작업 단위는 `T1`, `T2`처럼 task 번호로 표기한다. "Phase 1", "단계 1" 같은 명칭은 자율 루프 도구(pjc 등)의 내부 단계명과 혼동되므로 쓰지 않는다.
- 동일 계획이면 갱신, 새 기능이면 기존 내용 삭제 후 새로 작성한다. **기준 문서가 수정된 경우는 항상 새로 작성으로 본다.**

## 5단계: 완료 보고 형식
```md
## 완료 보고
- 수정 내용:
- 변경 파일:
- 검증 결과: (명령 + 결과. 미검증 시 "미검증" 명시)
- 문서 역대조 결과: (문서 기반 작업 시. 누락·잔존·변형 여부)
- 문서 갱신 내역: (생략 시 생략 사유)
- 남은 이슈:
- 승인 필요 항목:
```

- **축약 보고 허용 조건(전부 해당 시)**: 동작 변경 없음 + 신규 파일 없음 + 변경 라인 10줄 이하 → "수정 내용·변경 파일·검증 결과" 3항목만 보고할 수 있다 (문서 갱신 생략 조건과 동일 기준).

## Git 규칙
- 승인 없이 commit/push 금지.
- commit 메시지는 한글, `{유형}: {요약}` 형식 (유형: 기능/수정/리팩토링/문서/설정).
- commit 전 체크: 빌드 통과 + plan.md 체크박스 검증 + notes.md 갱신.
